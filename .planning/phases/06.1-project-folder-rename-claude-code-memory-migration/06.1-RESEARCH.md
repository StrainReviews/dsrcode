## 2. Technical Research: PowerShell Cmdlets (Plans 02-04)

### 2.1 Get-Process (prereq-check.ps1 — Plan 02)

**Source:** context7 `/microsoftdocs/powershell-docs` + exa ActiveDirectoryTools 2026-02-06 + Microsoft Learn 7.4/7.6

**Verified Syntax (PowerShell 5.1 + 7.4/7.6 compatible):**

```powershell
# By PID (our daemon check)
$daemon = Get-Process -Id 55564 -ErrorAction SilentlyContinue
if ($daemon) { $daemonPath = $daemon.Path }

# By Name (for claude.exe / Claude Code.exe processes)
$claudeProcs = Get-Process -Name "claude*" -ErrorAction SilentlyContinue

# Get all with details (CIM fallback for CommandLine info)
$cwdProcs = Get-CimInstance -ClassName Win32_Process -Filter "CommandLine LIKE '%cc-discord-presence%'"
```

**Critical Notes:**
- `Get-Process` does NOT expose `CommandLine` or current working directory — must use `Win32_Process` via `Get-CimInstance`.
- 32-bit PowerShell returns `$null` for `Path` when inspecting 64-bit processes. Always launch prereq-check.ps1 from 64-bit `pwsh.exe` or Windows PowerShell 5.1 x64.
- `-ErrorAction SilentlyContinue` is mandatory when checking a PID that might not exist.

### 2.2 Stop-Process (handoff.ps1 — Plan 03)

**Source:** context7 `/microsoftdocs/powershell-docs` query-docs

**Verified Syntax:**

```powershell
Stop-Process -Id 55564 -Force
# Or by process object:
Get-Process -Id 55564 -ErrorAction SilentlyContinue | Stop-Process -Force
```

**Critical Notes:**
- `-Force` bypasses confirmation on critical processes — required for non-interactive handoff.
- `Stop-Process` does not wait for the process to exit; `Wait-Process` or `Start-Sleep 2` may be needed before the subsequent `Rename-Item`.

### 2.3 Rename-Item (handoff.ps1 — Plan 03)

**Source:** exa `Netwrix 2024-10-09` + `Move-Item issue #25047` + SO #41365318

**Verified Syntax:**

```powershell
Rename-Item -Path "C:\Users\ktown\Projects\cc-discord-presence" -NewName "dsrcode" -ErrorAction Stop
```

**Critical Error Pattern (relevant to D-04 Handoff-Pause):**
- Rename-Item on a locked directory raises `IOException` ("File in use by another process"). SO #41365318 is the canonical reference: Windows does not permit deletion or renaming of a directory that is the cwd of any running process.
- `-ErrorAction Stop` converts the non-terminating error to a terminating error, allowing `try/catch` blocks to handle cleanup.
- PowerShell issue #25047 (2025-02): `Move-Item -Force` does not report "file in use" accurately — always prefer `Rename-Item` over `Move-Item` for this use case.

### 3.2 Plugin Cache Architecture (confirms D-08 + D-13)

From the live docs (2026-04-10 crawl):

> For security and verification purposes, Claude Code copies marketplace plugins to the user's local plugin cache (`~/.claude/plugins/cache`) rather than using them in-place. Each installed version is a separate directory in the cache. When you update or uninstall a plugin, the previous version directory is marked as orphaned and removed automatically 7 days later.

> The `${CLAUDE_PLUGIN_DATA}` directory resolves to `~/.claude/plugins/data/{id}/`, where `{id}` is the plugin identifier with characters outside `a-z, A-Z, 0-9, _, -` replaced by `-`. For a plugin installed as `formatter@my-marketplace`, the directory is `~/.claude/plugins/data/formatter-my-marketplace/`.

**Implication:** Our data dir is `~/.claude/plugins/data/dsrcode-dsrcode/`. Confirms `--keep-data` mandatory (D-12).

## 5. Sysinternals Handle.exe (Plan 02 — optional fallback)

**Source:** Referenced in SO #41365318 + CONTEXT.md canonical_refs

### 5.1 Usage Pattern

```powershell
# Check if handle.exe is available (optional dependency)
$handleExe = Get-Command handle.exe -ErrorAction SilentlyContinue

if ($handleExe) {
    # Invoke with -a (all types) -u (show username) and target path
    $handles = & handle.exe -a -u "C:\Users\ktown\Projects\cc-discord-presence" -nobanner 2>&1
    if ($handles -match "pid:\s*(\d+)") {
        Write-Host "Blocking PID: $($matches[1])"
    }
} else {
    # Fallback: scan for processes with CWD matching project dir
    $cimProcs = Get-CimInstance -ClassName Win32_Process |
                Where-Object { $_.CommandLine -like "*cc-discord-presence*" }
}
```

**Notes:**
- Handle.exe is optional; script must fall back gracefully.
- `-nobanner` suppresses the EULA banner when first-time run (2026 version auto-accepts after first run via registry).
- Running against a directory returns all descendants' handles.

## 6. Memory Directory Migration (Plan 03 — reuse from Discuss-Phase research)

**Source (cached from 06.1-DISCUSSION-LOG.md crawl4ai):** https://gist.github.com/gwpl/e0b78a711b4a6b2fc4b594c9b9fa2c4c (2026-03-26)

### 6.1 Canonical Pattern

```bash
# Git Bash or WSL:
cd ~/.claude/projects/
mv -- C--Users-ktown-Projects-cc-discord-presence C--Users-ktown-Projects-dsrcode
# Note: Use -- to prevent mv from interpreting the leading hyphen as an option flag
```

### 6.2 PowerShell Equivalent (our script)

```powershell
$oldMem = "$env:USERPROFILE\.claude\projects\C--Users-ktown-Projects-cc-discord-presence"
$newMem = "$env:USERPROFILE\.claude\projects\C--Users-ktown-Projects-dsrcode"
if (Test-Path $oldMem) {
    Rename-Item -Path $oldMem -NewName "C--Users-ktown-Projects-dsrcode" -ErrorAction Stop
}
```

**Why Rename-Item and not mv:** PowerShell is the target shell for handoff.ps1 (Windows-native, no bash dependency). `Rename-Item` handles hyphen-prefix paths correctly via `-Path` parameter (no `--` needed).

All in a single `Rename-Item` operation. Reversible via the inverse rename.
