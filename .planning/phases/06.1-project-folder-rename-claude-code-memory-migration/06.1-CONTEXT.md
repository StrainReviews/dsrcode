### Cluster 1: Folder Rename + Memory Migration

- **D-01 (Zielname LOCKED):** Projektordner-Rename-Ziel ist `dsrcode`. Neuer lokaler absoluter Pfad: `C:\Users\ktown\Projects\dsrcode`. GitHub Repo, Go module path (`github.com/StrainReviews/dsrcode`), plugin.json name field und marketplace ID sind bereits seit Phase 5 DIST-01/02/03 als `dsrcode` gesetzt — KEINE Änderung nötig.

- **D-02 (Memory Encoded Path LOCKED):** Neuer Claude Code memory directory-Name ist `C--Users-ktown-Projects-dsrcode`. Encoding-Algorithmus: absoluter Pfad mit `/`, `\` und `:` → `-` (bestätigt aus gwpl Gist 2026-03-26 + docs.anthropic.com/en/docs/claude-code/memory + existierender Ordnername).

- **D-03 (Kein Symlink / Kein Junction):** Nach dem Rename wird KEINE Rückwärtskompatibilitäts-Brücke vom alten Pfad zum neuen erstellt. Clean cut. Rationale: Claude Code resolvt Windows Junctions und Symlinks zum realpath, ein Symlink würde daher kein Memory-Aliasing bringen. Go module cache und IDE recent projects werden sowieso manuell aktualisiert. Konkrete Aktion: keine `mklink /J` oder `ln -s` Kommandos im Handoff.

- **D-04 (Handoff-Pause Pattern):** Phase 6.1 Plans werden gesplittet in Pre-Handoff (im alten Dir) und Post-Handoff (in neuer Session im neuen Dir). Plans 01-04 committen im alten Dir, dann pausiert die Phase mit HANDOFF.md Markdown-Checkliste. User führt den Filesystem-Rename manuell in externer Shell aus (NICHT in der laufenden Claude Code Session möglich wegen Windows file-locking constraint SO #41365318). User startet NEUE Claude Code Session im neuen Dir, dort läuft Plan 05 (Verification + Doc Updates + Summary).

- **D-05 (Memory Migration via atomar mv):** Migration-Command ist `cd ~/.claude/projects/ && mv -- C--Users-ktown-Projects-cc-discord-presence C--Users-ktown-Projects-dsrcode` (Git Bash) oder PowerShell-Equivalent `Rename-Item`. Single atomic operation, bewegt gleichzeitig `memory/` (MEMORY.md + individual auto-memory files), ~20 `*.jsonl` session transcripts und ~10 UUID/ subfolders. Offizielles Pattern aus gwpl Gist 2026-03-26 + Community-Workaround für #37954. Reversibel durch umgekehrtes `mv`.

- **D-06 (Prereq-Check Hybrid Pattern):** Vor dem Rename muss ein `prereq-check.ps1` (Plan 02) automatisch prüfen: (a) Status von dsrcode daemon PID 55564, (b) laufende Claude Code Prozesse mit cwd im Projekt-Dir, (c) offene File-Handles via Sysinternals Handle.exe (optional installed), (d) laufende go build/test Prozesse. Ergänzt durch manuelle Markdown-Checkliste (Teil von HANDOFF.md, Plan 03): IDE-Fenster schließen (VS Code, JetBrains), Git Bash Shells schließen, OneDrive sync pausieren falls involved.

- **D-07 (V3.1.10 Daemon Manual Kill):** Der aktuell installierte Plugin v3.1.10 hat KEINE auto-exit Logic (eingeführt erst in Phase 6 → v4.1.0). Daemon PID 55564 muss via `taskkill /f /pid 55564` oder `Stop-Process -Id 55564 -Force` manuell im Handoff beendet werden. Auto-exit grace period ist erst nach Plugin-Reinstall verfügbar.

## Specific Ideas

- **Handoff-Pause Pattern:** Phase 6.1 ist die erste GSD-Phase die eine HANDOFF-PAUSE nutzt — Plans 01-04 im alten Dir, User-Aktionen in externer Shell, Plan 05 in neuer Session. Das Pattern könnte für future Phasen relevant sein (z.B. large-scale refactoring mit IDE restart requirement).
- **Atomic mv für Memory-Migration:** Der offizielle Community-Workaround aus gwpl Gist ist elegant — ein einziger `mv -- OLD NEW` Command migriert gleichzeitig Auto-Memory + Session Transcripts + UUID subfolder ohne Inkonsistenz.
- **Version-Skip als Dokumentation-Strategie:** v4.0.0 skip ist bewusst — das ganze "Phase 5 → v4.0.0 Release" war ein internes Milestone, das public Release ist v4.1.0. CHANGELOG.md dokumentiert das bereits.
- **--keep-data als Safety-Default:** Selbst wenn der data dir leer ist, ist `--keep-data` das korrekte Pattern für production plugin reinstalls. Phase 6.1 etabliert das als Konvention.
- **Plugin Cache Bugs in einem Durchgang adressieren:** Phase 6.1 Handoff trifft ALLE 5 bekannten Plugin-Cache-Bugs (#41885 #19197 #29071 #37172 #29074) mit EINEM konsistenten Workaround-Chain. Das ist die "offizielle" inoffizielle Lösung aus der Community.
- **Sysinternals Handle.exe als optional Dependency:** prereq-check.ps1 funktioniert auch ohne Handle.exe installed (falls nicht), nutzt dann alternativ `Get-Process | Where-Object { $_.Path -like "*cc-discord-presence*" }` als Fallback.
